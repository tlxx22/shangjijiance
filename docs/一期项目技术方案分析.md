## 一、项目概述

### 1.1 项目定位

一期项目是一个基于 **Scrapy + Selenium + AI后处理** 的企业级招投标信息采集系统，目标是从39个央企/国企招投标平台自动采集、解析、筛选招标公告，并通过AI提取结构化数据入库。

### 1.2 技术栈

| 组件     | 技术选型          | 版本/说明            |
| -------- | ----------------- | -------------------- |
| 爬虫框架 | Scrapy            | 工业级爬虫框架       |
| 动态渲染 | Selenium + Chrome | 处理JavaScript页面   |
| AI提取   | DeepSeek API      | 从爬取数据中提取字段 |
| 数据库   | MySQL / Oracle    | 双数据库支持         |
| 任务调度 | APScheduler       | 定时任务             |
| 日志系统 | Loguru            | 统一日志管理         |

## 二、系统架构

详细见[爬虫流程图](https://yaowuteam.yuque.com/skl4ut/kaf115/kxa9p7pybgp2xxlz). 

**数据流转**

```plain
招投标网站 → Scrapy请求 → Selenium动态渲染(可选) → Spider解析
    → Pipeline处理 → AI结构化提取 → 数据入库 → 后续业务(匹配/通知)
```

## 三、核心模块分析

### 3.1 BaseSpider 基础爬虫类

**文件位置**：`scrapers/spiders/base_spider.py`

**核心功能**：

- 统一日志管理（业务日志 + 统一日志）
- 日期范围过滤（支持14种日期格式解析）
- 请求缓存机制（内存 + 文件双重缓存）
- 关键词匹配器集成
- WebDriver生命周期管理
- 智能翻页跳转算法

**设计亮点**：

- 连续过期项目检测（`MAX_CONSECUTIVE_EXPIRED_ITEMS = 10`），避免无效翻页
- 智能翻页跳转算法 `calculate_page_jump()`
- 多环境WebDriver适配（Windows/Linux）

### 3.2 Pipeline 数据处理管道

**文件位置**：`scrapers/pipelines.py`

**管道链路**：

```plain
DataProcessingPipeline (数据预处理)
    ↓
TenderPipeline (招标数据清洗)
    ↓
AIProcessingPipeline (AI结构化提取) ← DeepSeek API
    ↓
AttachmentDownloadPipeline (附件下载)
    ↓
JsonWriterPipeline (JSON备份 + 数据库写入)
```

**关键点**：Pipeline **只处理** Spider 已提取好的 Item，**不参与网络请求**。

### 3.3 AI处理模块

**文件位置**：`processors/ai_processing.py`

**处理时机**：数据采集**完成后**，作为后处理环节

**提取字段**（23个标准字段）：

- projectname (项目名称)、projectId (项目编号)
- budget (预算金额)、bidamount (中标金额)
- buyer (采购单位)、lotCandidates.type=中标 的 candidates（中标单位）
- area/city/district (地区信息)
- ...

**局限性**：AI无法指导采集过程，只能处理已爬取的数据

### 3.4 Selenium中间件

**文件位置**：`scrapers/middlewares.py`

**反爬配置**：

```python
chrome_options.add_argument('--disable-blink-features=AutomationControlled')
chrome_options.add_experimental_option('excludeSwitches', ['enable-automation'])
```

**UA（User-Agent）作用**：

- HTTP请求头中的标识字段，告诉服务器"我是什么浏览器"
- 爬虫默认UA会暴露身份（如 `Scrapy/2.x`）
- 添加随机UA伪装成真实浏览器，降低被反爬检测概率

## 五、一期问题分析

### 5.1 架构层面问题

| 问题           | 严重程度 | 影响                                           |
| -------------- | -------- | ---------------------------------------------- |
| 选择器不够通用 | 高       | 靠硬编码实现, 网站改版即失效，39套选择器需维护 |
| AI后置处理     | 中       | 无法指导爬取，浪费无效请求                     |
| 反爬能力不足   | 中       | 无风控检测，无验证码处理                       |

#### 选择器不够通用（核心问题）

```python
# 每个Spider都需要硬编码选择器
def parse_list(self, response):
    items = response.css('div.list-item')  # 网站HTML改版即失效
    for item in items:
        title = item.css('a.title::text').get()
        date = item.css('span.date::text').get()
```

**影响**：

- 39个网站 = 39套选择器需要维护
- 网站HTML结构变化 → 选择器失效 → 数据采集中断
- 人工排查成本高，无法自动适应

### 5.2 扩展成本

**添加新网站流程**：

1. 分析目标网站结构
2. 编写专用Spider类
3. 调试CSS/XPath选择器
4. 处理分页逻辑
5. 测试并部署
