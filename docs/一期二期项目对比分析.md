## 一、项目概述对比

### 1.1 项目定位

| 维度     | 一期                                       | 二期                                   |
| -------- | ------------------------------------------ | -------------------------------------- |
| 核心目标 | 从39个央企/国企平台采集招投标信息          | 同左，但更注重智能化和可维护性         |
| 技术路线 | Scrapy框架 + Selenium + AI后处理           | browser-use AI Agent + LLM实时决策     |
| 设计理念 | 规则驱动（硬编码选择器）                   | AI驱动（LLM语义理解）                  |

| 组件         | 一期                    | 二期                     |
| ------------ | ----------------------- | ------------------------ |
| 核心框架     | Scrapy                  | browser-use              |
| 动态渲染     | Selenium + Chrome       | browser-use内置Playwright |

| 任务调度     | APScheduler             | asyncio异步              |

## 二、系统架构对比

### 2.1 数据流对比

**一期数据流**（规则驱动）：

```plain
定时任务启动 → Spider生成请求 → Middleware判断模式
    → HTTP/Selenium访问网站 → 返回HTML
    → Spider解析（CSS选择器硬编码）
    → Pipeline后处理（AI提取字段）
    → 数据入库
```

**二期数据流**（AI驱动）：

```plain
main.py启动 → 加载YAML配置 → AI Agent接管浏览器
    → LLM理解页面 → 实时筛选条目 → 点击进入详情
    → 截图 + AI提取字段（实时）
    → 保存结果
```

### 2.2 流程图

- 一期流程图：[09_universal_spider_flow.mmd](../../wdcl-opportunity-monitor-spider/mermaid/09_universal_spider_flow.mmd)
- 二期流程图：[二期项目流程图.mmd](二期项目流程图.mmd)

### 2.3 核心差异

| 环节       | 一期                           | 二期                           |
| ---------- | ------------------------------ | ------------------------------ |
| 页面访问   | 两种模式（HTTP/Selenium）      | 统一浏览器模式                 |
| 元素定位   | CSS/XPath选择器                | LLM语义理解                    |
| 筛选时机   | 爬取后过滤                     | 浏览时实时筛选                 |
| AI角色     | Pipeline后处理                 | Agent实时决策                  |
| 翻页逻辑   | 选择器提取"下一页"按钮         | LLM识别翻页按钮                |
| 风控处理   | 无                             | 自动检测并跳过                 |

## 四、问题解决对比

### 4.1 一期核心问题 → 二期解决方案

| 一期问题                   | 二期解决方案                     |
| -------------------------- | -------------------------------- |
| 39套选择器需人工维护       | LLM语义理解，无需维护选择器      |
| 网站改版导致选择器失效     | AI自动适应页面变化               |
| AI后置，无法指导爬取       | AI前置决策，边看边筛选           |
| 无风控检测                 | 自动检测风控/验证码，记录跳过    |
| 新增网站需写代码           | YAML配置即可，无需写代码         |
| 先爬后筛，浪费无效请求     | 实时筛选，精准爬取               |

### 4.2 扩展成本对比

**一期添加新网站**：

1. 分析目标网站结构（1-2小时）
2. 编写专用Spider类
3. 调试CSS/XPath选择器
4. 处理分页逻辑
5. 测试并部署

**预计工时**：1-3天/网站

**二期添加新网站**：

1. 在 `sites_config.yaml` 添加配置
2. 测试运行

**预计工时**：10-30分钟/网站

## 五、适用场景

### 5.1 一期适用场景

- 目标网站结构稳定、长期不变
- 需要大规模并发采集（Scrapy生态成熟）
- 对采集速度要求极高
- 已有成熟的选择器维护团队

### 5.2 二期适用场景

- 目标网站多、结构各异
- 网站频繁改版
- 需要快速接入新网站
- 重视开发效率和维护成本
- 团队规模小，无专人维护选择器

## 六、总结

### 6.1 架构演进

```plain
一期：规则驱动 → 硬编码选择器 → AI后处理
二期：AI驱动 → LLM语义理解 → 实时决策
```

### 6.2 核心价值

| 维度         | 一期                 | 二期                 |
| ------------ | -------------------- | -------------------- |
| 开发效率     | 低（需写选择器）     | 高（配置即用）       |
| 维护成本     | 高（39套选择器）     | 低（无选择器）       |
| 扩展性       | 差（每站写代码）     | 好（YAML配置）       |
| 智能程度     | 低（规则匹配）       | 高（语义理解）       |
| 运行成本     | 低（无LLM调用）      | 中（LLM API费用）    |
| 采集速度     | 快                   | 中等                 |

### 6.3 技术趋势

二期项目代表了爬虫技术的演进方向：

- **从规则到智能**：用AI理解代替硬编码规则
- **从后处理到实时**：AI从后处理环节前移到决策核心
- **从代码到配置**：降低技术门槛，提高可维护性
